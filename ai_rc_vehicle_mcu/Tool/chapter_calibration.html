1. 배터리 전압 모니터링 (ADC)

1.1 왜 배터리 전압을 측정해야 하는가

RC카에 사용되는 7.4V 리튬이온 배터리(2S)는 만충 시 8.4V, 공칭 7.4V, 방전 한계 약 6.0V의 전압 범위를 가집니다. 주행 중 전압이 지속적으로 하락하면 모터 출력이 감소하고, 과방전 시에는 배터리 수명이 영구적으로 손상됩니다.

따라서 MCU에서 배터리 전압을 실시간으로 읽어 (1) 전압 보상 제어와 (2) 저전압 경고에 활용합니다.

1.2 전압 분배 회로 (Voltage Divider)

TC237의 ADC 입력 허용 전압은 최대 5V입니다. 리튬이온 배터리 전압(최대 8.4V)을 직접 인가하면 ADC가 손상되므로, 저항 분압 회로를 사용하여 전압을 절반으로 낮춥니다.

V_battery (7.4V) | R1 = 10kΩ | +---------> TC237 ADC (AN0, P40.0) | R2 = 10kΩ | GND
Vadc = Vbattery × R2 / (R1 + R2) = Vbattery × 0.5
Vbattery = 8.4V → Vadc = 4.2V (ADC 안전 범위 이내)

1.3 ADC 값에서 실제 전압으로 변환

TC237 VADC는 12비트 분해능(0~4095)을 가지며, 기준 전압(Vref)은 5.0V입니다. ADC raw 값에서 실제 배터리 전압을 구하는 수식은 다음과 같습니다:

Vbattery(mV) = ADCraw × 5000 / 4095 × 2
= ADCraw × 10000 / 4095

분압비 0.5의 역수(×2)를 곱하여 원래 배터리 전압을 복원합니다.

소프트웨어 구현

uint16 DrvAdc_GetBatteryMv(void)
{
    Ifx_VADC_RES result;
    result = IfxVadc_Adc_getResult(&s_adcChBattery);

    if (result.B.VF)
    {
        /* V_battery_mV = raw * (5 * g_batMultiplier) / 4095
         * g_batMultiplier = 2000 (x2.000)               */
        return (uint16)((uint32)result.B.RESULT
                        * (5u * g_batMultiplier) / 4095u);
    }
    return 0;
}
캘리브레이션 계수 (g_batMultiplier): 이론적 분압비는 정확히 2.0이지만, ADC 입력 임피던스와 저항 오차로 인해 실측값과 차이가 발생할 수 있습니다. 이 경우 멀티미터 실측값과 비교하여 보정 계수를 조정합니다. 예: 실측 8.0V인데 패널에 7.52V로 표시되면, 계수를 2000 → 2128로 조정합니다.

1.4 텔레메트리 전송

100ms 주기의 태스크에서 배터리 전압을 포함한 센서 데이터를 UART로 전송합니다:

L:1234,R:2345,U:30,B:7400\r\n
                      ^^^^
                      배터리 7400mV = 7.4V

패널(PC)에서는 이 텍스트를 파싱하여 실시간 전압 그래프와 숫자 표시를 합니다.

2. 모터 캘리브레이션 (DFLASH 저장)

2.1 왜 모터별 캘리브레이션이 필요한가

4WD RC카의 4개 모터는 개체 차이(내부 저항, 마찰, 기어 편차)로 인해 동일한 Duty를 인가해도 실제 회전 속도가 다릅니다. 이로 인해 직진 시 한쪽으로 쏠리는 현상이 발생합니다.

엔코더가 없는 오픈루프 시스템에서는 각 모터의 Duty를 개별적으로 보정하여 직진성을 확보해야 합니다.

2.2 캘리브레이션 데이터 구조

캘리브레이션 값은 TC237의 DFLASH(Data Flash, 0xAF000000)에 8바이트(1페이지) 단위로 저장됩니다.

바이트필드범위기본값설명
[0]dutyFL0~100%70전좌(Front-Left) 모터 Duty
[1]dutyFR0~100%70전우(Front-Right) 모터 Duty
[2]dutyRL0~100%70후좌(Rear-Left) 모터 Duty
[3]dutyRR0~100%70후우(Rear-Right) 모터 Duty
[4]turnFront0~100%100회전 시 전륜 속도 팩터
[5]turnRear0~100%100회전 시 후륜 속도 팩터
[6]magicLo0xFE-유효성 검증 매직넘버
[7]magicHi0xCA-
매직넘버(0xCAFE): DFLASH가 비어있거나 데이터가 손상된 경우를 감지합니다. 부팅 시 매직넘버가 일치하지 않으면 기본값(70%)을 사용합니다.

2.3 DFLASH Write 시퀀스

TC237 DFLASH는 NOR Flash 방식으로, 쓰기 전에 반드시 섹터 단위 삭제(Erase)가 필요합니다. 한 번 0→1로 바꿀 수 없고, Erase로 전체를 1(0xFF)로 만든 후에만 Write가 가능합니다.

[Save 시퀀스]
1. Endinit 해제         -- Flash 레지스터 보호 해제
2. IfxFlash_clearStatus -- 이전 에러 플래그 클리어
3. IfxFlash_eraseSector -- Sector 0 전체 삭제 (8KB)
4. IfxFlash_waitUnbusy  -- 삭제 완료 대기
5. IfxFlash_enterPageMode -- 쓰기 모드 진입
6. IfxFlash_loadPage2X32  -- 8바이트 데이터 로드
7. IfxFlash_writePage     -- 페이지 쓰기 실행
8. IfxFlash_waitUnbusy    -- 쓰기 완료 대기
9. Endinit 복원          -- Flash 레지스터 보호 복원
10. 검증: 매직넘버 읽어서 성공 확인
주의: Erase 후 검증을 반드시 수행해야 합니다. 간헐적으로 Erase가 실패하는 경우가 있어, 본 구현에서는 최대 3회 재시도 로직을 포함합니다. DFLASH의 Erase/Write 수명은 약 10만 회이므로 캘리브레이션 용도로는 충분합니다.

2.4 캘리브레이션 모델

각 모터의 실제 Duty는 다음 수식으로 결정됩니다:

Dutyactual = (Calmotor + BatComp) × Speed / 100
변수설명범위
Calmotor각 모터별 캘리브레이션 Duty (DFLASH 저장)0~100%
BatComp배터리 전압 보상값 (런타임 자동 계산)0~8
Speed조종기에서 전송된 속도 명령0~100%

예시

파라미터FLFRRLRR
Cal (저장값)70707575
BatComp (7.4V)+4+4+4+4
Speed (100%)100
실제 Duty74%74%79%79%

후륜(RL/RR)이 전륜(FL/FR)보다 높은 Duty를 가지는 것은, 차량 무게 중심이 뒤쪽에 있어 후륜에 더 많은 토크가 필요하기 때문입니다.

2.5 실시간 캘리브레이션 절차

본 시스템의 특징은 주행 중 실시간 조절이 가능하다는 점입니다. 기존 방식처럼 "주행→측정→수정→재주행"을 반복할 필요 없이, 달리면서 바로 보정합니다.

  1. PC 패널 연결 → LIVE 모드 ON
  2. W키로 전진 주행 시작
  3. 차가 오른쪽으로 쏠리면 → FR/RR의 − 버튼 클릭
  4. 차가 왼쪽으로 쏠리면 → FL/RL의 − 버튼 클릭
  5. 값 변경 즉시 MCU에 반영 — 주행하면서 직진성 확인
  6. 직진 확인 후 SAVE → DFLASH에 영구 저장
  7. 재부팅해도 자동 복원 (부팅 시 LoadCalibration 호출)
MCU RAM 모니터링: 패널 하단에 "MCU RAM" 행이 있어 MCU에 실제 적용된 캘리브레이션 값을 확인할 수 있습니다. SEND 후 자동으로 CAL_QUERY 명령을 보내 현재 값을 읽어옵니다.

3. 배터리 전압 보상 제어

3.1 문제: 전압 하락에 따른 속도 감소

RC카를 5분쯤 신나게 주행하면, 어느 순간 "느려졌다"는 느낌이 듭니다. 같은 조종 명령을 줬는데 처음보다 속도가 명백히 줄어든 것입니다.

이는 배터리 전압이 떨어졌기 때문입니다. DC 모터의 RPM은 인가된 평균 전압에 거의 비례합니다:

RPM ∝ Vavg = Vbattery × Duty / 100

배터리가 8.4V에서 6.4V로 떨어지면, 같은 Duty 70%에서:

배터리 전압Duty평균 전압속도 비율
8.4V (만충)70%5.88V100%
7.4V70%5.18V88%
6.4V (방전)70%4.48V76%

약 24%의 속도 감소가 발생합니다.

3.2 해결: 전압 보상 (Voltage Compensation)

이상적인 해결 방법은 엔코더를 사용한 피드백 제어(PID)이지만, 본 시스템에서는 엔코더가 없습니다. 대신 배터리 전압을 ADC로 읽어, 전압이 떨어진 만큼 Duty를 올려주는 전압 보상(Feedforward) 방식을 사용합니다.

BatComp = (Vref − Vcurrent) / (Vref − Vmin) × COMPmax

Vref = 8400mV (만충),   Vmin = 6400mV (최저),   COMPmax = 8

보상 테이블

배터리 전압보상값Cal=70일 때 실제 Duty평균 전압 (보상 후)
8.4V (만충)+070%5.88V
7.9V+272%5.69V
7.4V+474%5.48V
6.9V+676%5.24V
6.4V (최저)+878%4.99V

완벽한 보상은 아니지만, 보상 없이 4.48V까지 떨어지는 것에 비해 4.99V를 유지하므로 체감 속도 변화를 크게 줄일 수 있습니다.

소프트웨어 구현

#define BAT_VREF_MV   8400u   /* 만충 전압 기준 */
#define BAT_VMIN_MV   6400u   /* 최저 전압 */
#define BAT_COMP_MAX  8       /* 최대 보상 Duty */

static sint16 getBatCompensation(void)
{
    uint16 batMv = DrvAdc_GetBatteryMv();

    if (batMv == 0u || batMv >= BAT_VREF_MV)
        return 0;              /* 만충 or 미측정 */

    if (batMv <= BAT_VMIN_MV)
        return BAT_COMP_MAX;   /* 최저 이하 → 최대 보상 */

    /* 선형 보간 */
    return (sint16)(((uint32)(BAT_VREF_MV - batMv) * BAT_COMP_MAX)
                    / (BAT_VREF_MV - BAT_VMIN_MV));
}
피드백 vs 피드포워드: 엔코더 기반 PID 제어(피드백)는 목표 속도를 정확히 추종하지만 센서가 필요합니다. 전압 보상(피드포워드)은 센서 없이 ADC만으로 구현 가능하며, 모터 부하 변화(경사, 노면)에는 대응하지 못하지만 배터리 방전에 의한 점진적 속도 저하는 효과적으로 보상합니다.

4. 저전압 경고 알람 (부저)

4.1 설계 목적

리튬이온 배터리는 과방전 시 비가역적 손상이 발생합니다. 셀당 3.0V(2S 기준 6.0V) 이하로 떨어지면 배터리 수명이 영구적으로 단축됩니다. 이를 방지하기 위해, 전압이 임계값(6.8V) 이하로 떨어지면 부저를 울려 사용자에게 경고합니다.

4.2 하드웨어 구성

패시브 부저(Passive Buzzer)를 사용합니다. 패시브 부저는 외부에서 PWM 신호를 공급해야 소리가 나며, PWM 주파수를 변경하면 음높이(피치)가 바뀌므로 멜로디 재생이 가능합니다.

항목사양
부저 종류패시브 부저 (Arduino 호환)
MCU 출력TOM0_Ch10 → TOUT2 → P02.2 (X103 핀 15)
PWM 주파수262Hz ~ 880Hz (음계에 따라 가변)
PWM Duty50% (ON) / 0% (OFF)

4.3 음계 제어 원리

패시브 부저의 음높이는 PWM 주파수로 결정됩니다. GTM TOM 타이머의 주기(Period)를 변경하면 주파수가 바뀝니다:

Period = FXCLK1 / Frequency
FXCLK1 = 6,250,000 Hz

예: 440Hz(A4) → Period = 6,250,000 / 440 = 14,205 ticks 
void DrvBuzzer_PlayNote(uint16 freqHz)
{
    Ifx_TimerValue newPeriod = 6250000u / freqHz;
    Ifx_TimerValue trigger   = newPeriod / 2;  /* 50% duty */

    IfxGtm_Tom_Timer_setPeriod(&s_tomBuzzer, newPeriod);
    IfxGtm_Tom_Timer_setTrigger(&s_tomBuzzer, trigger);
}

4.4 멜로디 재생 구조

100ms 주기 태스크에서 배터리 전압을 확인하고, 임계값(6.8V) 미만이면 미리 정의된 멜로디 배열을 순서대로 재생합니다.

/* Super Mario Bros Theme — 주파수(Hz), 0=무음 */
static const uint16 s_lowBatMelody[] = {

    784, 0, 0, 0, 392, 0, 0, 0,         /* G5 . . . G4 . . .   */
    523, 0, 0, 392, 0, 0, 330, 0,       /* C5 . . G4 . . E4 .   */
    0, 440, 0, 494, 0, 466, 440, 0,     /* . A4 . B4 . Bb4 A4 . */
    392, 659, 0, 784, 880, 0, 698, 784, /* G4 E5 . G5 A5 . F5 G5 */
    0, 659, 0, 523, 587, 494, 0, 0,     /* . E5 . C5 D5 B4 . .   */

재생 로직

void AppTask_100ms(void)
{
    uint16 batMv = DrvAdc_GetBatteryMv();

    if ((batMv > 0u) && (batMv < 6800u))   /* 6.8V 미만 */
    {
        uint16 note = s_lowBatMelody[s_melodyIdx];
        if (note > 0u)
            DrvBuzzer_PlayNote(note);   /* 음 재생 */
        else
            DrvBuzzer_Off();            /* 무음 */

        s_melodyIdx++;
        if (s_melodyIdx >= MELODY_LEN)
            s_melodyIdx = 0u;           /* 반복 재생 */
    }
    else
    {
        DrvBuzzer_Off();
        s_melodyIdx = 0u;
    }
}

왜 마리오 테마인가? 단순한 "삐-삐-삐" 경고음 대신 인식하기 쉬운 멜로디를 사용하면 (1) 배터리 부족 상황을 즉시 인지할 수 있고, (2) 주변 소음과 구별이 용이합니다. 또한 패시브 부저의 가변 주파수 기능을 활용한 응용 사례로서 교육적 가치가 있습니다.
